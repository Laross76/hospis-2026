## 7. Виды тестирования
- Исследовательское тестирование:
  
 В отсутствие чётких требований может быть полезно провести исследовательское тестирование, чтобы лучше понять приложение и его поведение.
   
- Функциональное тестирование:
  
Важно проверить, работают ли все функции (например, авторизация, создание и редактирование новостей, навигация между разделами и т.д.) так, как задумано. Это позволит выявить любые несоответствия между ожидаемыми и фактическими результатами.

- Тестирование пользовательского интерфейса (UI Testing):
  
Проверки связанные с удобством использования и отображением интерфейса (например, адаптация под разные размеры экранов, правильное отображение текста и элементов интерфейса). UI тестирование поможет убедиться, что интерфейс интуитивно понятен и доступен для пользователей.

- Тестирование на удобство использования (Usability Testing):
  
Этот вид тестирования важен для оценки общего взаимодействия пользователя с приложением. Проверяя удобство навигации и доступность функций, можно сделать выводы о том, насколько приложение соответствует потребностям пользователей и как они реагируют на его интерфейс.

## 8. Техники тест-дизайна
- Эквивалентное разбиение (Equivalence Partitioning)

Разделение входных данных на классы эквивалентности (валидные/невалидные) например авторизация, фильтрация, ввод данных.

- Анализ граничных значений (Boundary Value Analysis)

Тестирование значений на границах диапазонов (например даты) фильтрация по датам, ввод спецсимволов.

- Сттейкхолдерное тестирование (Stakeholder Analysis)

Определение ожиданий пользователей от функций (например, UI/UX). Аккордеоны, навигация.

- Проверка состояний (State Transition Testing)

Переходы между экранами/состояниями приложения. Навигация, отмена действий.

## 11. Перечень используемых инструментов

- Android Studio - официальная интегрированная среда разработки основана на IntelliJ IDEA для создания приложений на платформе Android, которая предлагает множество инструментов и функций.
  
- Эмулятор Android API 37 - соответствует Android 17.
  
- Espresso - фреймворк для автоматизированного тестирования пользовательского интерфейса мобильных приложений на платформе Android. 
  
- Java JDK 17 — это комплект разработчика приложений на платформе Java SE с долгосрочной поддержкой (LTS). Он включает компилятор, стандартные библиотеки, среду выполнения (JRE) и инструменты для создания программ.
  
- JUnit 4 — тестовый фреймворк для Java, предназначенный для упрощения процесса тестирования программного обеспечения. Он предоставляет инструменты и аннотации, которые позволяют легко писать и выполнять автоматизированные тесты.
  
- Record Espresso Test - инструмент, который автоматически записывает действия пользователя для создания тестов.
  
- Layout Inspector - это встроенный инструмент разработки в Android Studio, который позволяет просматривать, анализировать и отлаживать пользовательский интерфейс приложения в реальном времени. Он визуализирует иерархию компонентов (View) и помогает быстро находить ошибки верстки, перекрытия и проблемы с производительностью.
  
- UI Automator - это официальный инструмент из Android SDK для создания автоматизированных тестов пользовательского интерфейса. Он позволяет имитировать действия пользователя (клики, свайпы, ввод текста) на мобильном устройстве и проверять реакцию системы.
  
- GitHub - это крупнейшая облачная платформа для хостинга IT-проектов, совместной разработки и общения. Она работает на базе системы контроля версий Git, позволяя тысячам программистов со всего мира безопасно и эффективно работать над общим кодом в одном месте.
  
- Allure – гибкий инструмент для составления отчетов о тестировании, который показывает понятное представление того, что было протестировано, в форме веб-отчета.

## 13. Перечень и описание возможных рисков при автоматизации
- Отсутствие Требований к тестированию

Описание:

Без наличия четких требований к тестированию, тестирование превращается в импровизацию (Ad-hoc), где инженер проверяет систему на основе личной интуиции, а не ожидаемых результатов, что приводит к критическим рискам.

Последствия:

Непредсказуемый результат: невозможно определить, работает система правильно или нет, так как нет эталона для сравнения.

Пробелы в покрытии: тестируется только видимый функционал, пропуская важные нефункциональные требования (безопасность, производительность, интеграции).

- Отсутствие Технического Задания (ТЗ)

Описание:

Отсутствие технического задания (ТЗ) приводит к неэффективному тестированию, пропуску дефектов и задержкам в проекте.

Последствия:

Пропускаются критические дефекты.

Тратится время на проверку всего, а не ключевых сценариев.

Выпускаете ПО с дефектами из-за невыявленных требований.

Нет стабильных наборов проверок, тесты зависят от текущего понимания.
